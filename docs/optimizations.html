Speed Optimizations — What Each One Does

Five techniques make training and preprocessing faster without changing the loss math or the architecture. Here is what each one actually does, and why the results stay the same.

① Mixed precision (float16 + loss scaling)

~1.4–2.0× faster

What it does: runs the heavy matrix-multiplies and convolutions in 16-bit floats instead of 32-bit. Modern GPUs (your RTX 5060 included) have tensor cores that crunch float16 roughly twice as fast and use half the memory bandwidth.

how it stays accurate

The fragile parts are kept in float32: the generator's tanh output, the discriminator's logits, and the attention softmax. A loss scaler multiplies the loss up before computing gradients (so tiny float16 gradients don't vanish to zero) and divides them back down before the optimizer step. Same numbers, just stored more efficiently.

Accuracy-safe: the optimizer, learning rates and gradients are identical — only the storage format of intermediate tensors changed.

code

tf.keras.mixed_precision.set_global_policy("mixed_float16")
opt = mixed_precision.LossScaleOptimizer(Adam(lr))   # auto loss scaling

② tf.data pipeline + prefetch

~1.05–1.25× faster

What it does: while the GPU trains on batch N, the CPU already prepares and copies batch N+1. The old code did this serially — build a tensor, copy it to the GPU, wait, then train — so the GPU sat idle during every copy.

OLD — copy then wait, every step

for step in range(steps):
    r = tf.constant(imgs[s:e])   # H→D copy, blocks
    lbs = tf.one_hot(...)        # on host
    train_step(r, lbs)           # GPU waited

NEW — overlap copy with compute

ds = (Dataset.from_tensor_slices(idx)
        .shuffle(N).batch(B)
        .map(gather_fn)          # on CPU workers
        .prefetch(AUTOTUNE))     # next batch ready
Accuracy-safe: exact same images, same batch size, same shuffle — only the timing of the copies changed.

③ In-graph one-hot + vectorized targets

removes per-step Python overhead

What it does: moves small repeated operations off slow Python and into the compiled graph. Two specific wins:

• One-hot labels are built inside the pipeline instead of on the host every step.

• The Phase-2 structural target (a random real image per sample) was chosen with a 32-element Python loop calling np.random.choice every step. Now the whole epoch's targets are picked with one vectorized NumPy expression and gathered in-graph.

OLD — Python loop per step

tgt = [imgs[np.random.choice(
         members[c])] for c in lbls]
tgt = np.stack(tgt)          # 32× choice / step

NEW — vectorized once per epoch

rnd = (rng.random(N)*counts[lbl])
idx = flat_members[off[lbl]+rnd]  # one shot
target_idx_var.assign(idx)
Accuracy-safe: identical selection rule (prototype in Phase 1, random real member in Phase 2) — just computed in bulk instead of one element at a time.

④ MediaPipe instance reuse (Model B only)

~5–15× faster extraction

What it does: builds the MediaPipe hand detector once and reuses it for all 54k images. The old code created a brand-new detector inside a with block on every call — and the cascade tries up to 4 strategies per image, so it loaded the model graph up to 4 times per image. Loading the model is far more expensive than actually running it.

OLD — rebuild detector every call

def run(img):
    with mp.Hands(...) as h:  # loads graph!
        return h.process(img)
# ~200k model loads over 54k imgs

NEW — one detector, reused

with HandLandmarkExtractor() as ext:
    for im in images:
        ext.extract(im)   # reuses Hands
Accuracy-safe: identical detector, identical confidence thresholds and cascade — the landmarks produced are exactly the same, only the wasteful re-loading is gone.

⑤ XLA compilation (optional, off by default)

+1.1–1.5× extra

What it does: XLA fuses many small GPU operations into fewer, larger compiled kernels, cutting launch overhead and memory traffic. Enabled with one flag on the training step.

It is off by default only because compilation can occasionally be picky about specific GPUs/op combinations — not because it affects results. Flip USE_XLA = True in config.py and benchmark; if it compiles, keep it.

Accuracy-safe: numerically equivalent — it only reorganizes how the same operations are executed.
